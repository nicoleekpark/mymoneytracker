## 개요

**제품명**: HoH Finance Tracker  
**작업용 이름**: 변경 가능

### 제품 설명

HoH Finance Tracker는 **수동적이 아닌, 의도적인 금융 기록**을 목표로 설계된 크로스플랫폼 개인 금융 앱이다  
자동 은행 연동 앱과 달리, HoH는 사용자가 모든 거래를 직접 기록하도록 유도하여 소비 인식과 통제력을 높인다

**핵심 철학**:
- **오프라인 우선**: 모든 데이터는 expo-sqlite를 통해 로컬에 저장
- **인증 불필요**: 진입 장벽 최소화, 즉시 사용 가능
- **프라이버시 중심**: 사용자 데이터는 기기 밖으로 나가지 않음 (v1)
- **Google Calendar에서 영감을 받은 UX**: 날짜 중심 탐색과 명확한 시간 흐름

## 제품 비전

### 문제 정의

많은 사람들이 금융 인식에 어려움을 겪는 이유는 도구 부족이 아니라 다음과 같다
- 자동 추적은 관찰만 남기고, 참여를 유도하지 못함
- 복잡한 기능은 단순한 명확성을 원하는 사용자에게 부담
- 은행 연동은 신뢰와 계정 연결이라는 마찰을 발생
- 기존 앱은 지출 패턴을 직관적으로 보여주지 못함

### 해결책

HoH Finance Tracker는 다음을 제공한다
1. **시각적 명확성**: Google Calendar 스타일의 일별 현금 흐름
2. **수동적 기록이 아닌 의도적 기록**: 직접 입력을 통한 인식 강화
3. **즉시 시작**: 가입 없음, 은행 연결 없음, 학습 곡선 없음
4. **크로스플랫폼**: iOS, Android, Web에서 동일한 경험

```
│                                                  │
│              [Content by Mode]                   │
│                                                  │
└──────────────────────────────────────────────────┘
```

#### 1.1 Overview 모드

##### 월간 뷰 (기본)

**예산 & 지출 카드**
- 예산 대비 실제 지출 표시
- 진행 바 (예산 이내 초록, 초과 시 빨강)
- 남은 금액 또는 초과 금액 표시
- 예산 초과 날짜 표시

**일별 현금 흐름 캘린더**
- 일–토 7일 그리드
- 각 날짜 셀에:
  - 날짜 ('오늘'에 하이라이트)
  - 거래 수 배지
  - 지출 합계 (빨강)
  - 수입 합계 (초록)
- 토글: 지출 | 수입 (최소 하나는 활성)
- 날짜 탭 → 해당 날짜 거래 목록

**월간 카테고리별 지출**
- 상위 5개 + 기타 도넛 차트
- 중앙: 월 총 지출
- 하단 리스트:
  - 색상 점
  - 카테고리명
  - 금액
  - 비율
- 향후: 카테고리 탭 → 상세 거래

##### 연간 뷰

- 연간 수입/지출 전체 개요
- 카테고리/서브카테고리 별 분석
- 월별 세부 분석
- 연간 총합 및 월 평균
- 카테고리별 바 차트
- 하위 카테고리 확장 가능

**예산과 소비**
- 연간 예산과 실제 소비 비교 (세팅되어 있는 경우)
- 달별 비교 분석

**연간 카테고리 별 소비**
- 바 차트: 카테고리 별 금액 순으로 정렬 (내림차순)
- 각 바는 확장 가능:
  - "Expand list" 버튼 누르면 모든 서브 카테고리 제공

##### All View

**카테고리별 지출**:
- 기록된 전체 기간(lifetime) 동안의 누적 지출
- 총 금액 기준 내림차순으로 카테고리 정렬
- Year 뷰와 동일한 확장(Expand) 패턴

#### 1.2 Assets Tab

**상태**: TBD (v1에서는 플레이스홀더)
**향후 목적**: 비현금 자산(부동산, 투자 등) 추적

**계정별 카드**:

### 2. 거래 추가 (+ 버튼)

**모달 패턴**: Apple Calendar에서 영감

```
┌──────────────────────────────────────────────────┐
│  [Cancel]      Add Transaction            [Save] │
├──────────────────────────────────────────────────┤
│  [Expense] [Income] [Transfer]                   │ ← Type Toggle
├──────────────────────────────────────────────────┤
│  Item: _________________________________________ │
│  Merchant: _____________________________________ │
│  Amount:          $___________          [Keypad] │
│  Date:            Jan 29, 2026                   │
│  Account:         Chase Checking         ▼       │
│  Category:        Food & Dining          ▼       │
│  Subcategory:     Restaurants            ▼       │
│  Note:            _______________________        │
└──────────────────────────────────────────────────┘
```

#### 거래 유형

**지출(Expense)**:
- 항목/가맹점(텍스트)
- 금액(숫자 키패드, 센트 단위로 저장)
- 날짜(데이트 피커)
- 계정(드롭다운)
- 카테고리 & 하위카테고리(계층형 피커)
- 메모(선택 텍스트)

**수입(Income)**:
- 항목/소스(텍스트)
- 금액(숫자 키패드)
- 날짜(데이트 피커)
- 계정(드롭다운)
- 메모(선택 텍스트)

**이체(Transfer)**:
- 항목(텍스트)
- 금액(숫자 키패드)
- 출금 계정(드롭다운)
- 입금 계정(드롭다운)
- 날짜(데이트 피커)
- 메모(선택 텍스트)

**UX 요구사항**:
- 금액 키패드: 계산기 스타일, 항상 표시
- 기본 날짜: 오늘
- 기본 계정: 마지막으로 사용한 계정
- 저장 검증: 금액 > 0, 계정 선택 필수
- 성공 피드백: 짧은 토스트 + 모달 닫기

## Technical Architecture

### Platforms & Tech Stack

**플랫폼**:
- iOS (우선)
- Android (동일 기능 제공)
- Web (모바일 우선, 동일 기능 제공)

**프론트엔드**:
- **프레임워크**: Expo (SDK 52+)
- **UI 라이브러리**: React Native + Tamagui
- **언어**: TypeScript
- **라우팅**: Expo Router (파일 기반)
- **상태 관리**: Zustand

**영속성(Persistence)**:
- **DB**: expo-sqlite (기기 내 SQLite)
- **마이그레이션 시스템**: 커스텀 마이그레이션 러너
- **시드 데이터**: 개발용 JSON fixtures

**아키텍처 패턴**:

```
┌─────────────────────────────────────────────┐
│         Presentation Layer                  │
│  (features/, app/, shared/components)       │
├─────────────────────────────────────────────┤
│          Domain Layer                       │
│  (domain/ - Pure business logic)            │
├─────────────────────────────────────────────┤
│       Infrastructure Layer                  │
│  (infrastructure/ - DB, repos, mappers)     │
└─────────────────────────────────────────────┘
```


**핵심 원칙**:
- 오프라인 우선 (네트워크 의존 없음)
- 결정론적 계산 (부동소수점 금액 사용 금지) - 센트로 저장
- 타입 안전한 도메인 모델
- 데이터 접근을 위한 리포지토리 패턴
- 관심사 분리(Separation of concerns)

## Edge Cases & Data Requirements

### 금융 엣지 케이스

| 케이스 | 처리 |
|------|------|
| **음수 금액** | 입력 불가, 에러 표시 |
| **0원 금액** | 입력 불가, 에러 표시 |
| **미래 날짜 거래** | 허용 (예정 지출) |
| **중복 거래** | 자동 중복 제거 없음, 사용자 책임 |
| **통화 정밀도** | 센트 단위 정수로 저장, 2자리 소수로 표시 |
| **큰 금액** | 상한 없음, UI에서 K/M 축약 표기 |
| **신용카드 결제** | 이체 타입 사용 (자산 → 부채 계정) |
| **환불** | 음수 지출 또는 양수 수입으로 입력 |

### 데이터 요구사항

**거래(Transaction) 필드**:
- `id` (UUID, 필수)
- `type` (expense | income | transfer, 필수)
- `amount_cents` (정수 > 0, 필수)
- `occurred_at` (ISO 날짜-시간, 필수)
- `account_id` (UUID, 필수)
- `category_id` (UUID, 수입/이체는 선택)
- `subcategory_id` (UUID, 선택)
- `item` (문자열, 필수, 최대 200자)
- `merchant` (문자열, 선택, 최대 200자)
- `note` (문자열, 선택, 최대 500자)
- `created_at` (ISO 날짜-시간, 필수)
- `updated_at` (ISO 날짜-시간, 필수)

**계정(Account) 필드**:
- `id` (UUID, 필수)
- `name` (문자열, 필수, 최대 100자)
- `nature` (asset | liability, 필수)
- `kind` (cash | checking | savings | credit_card | loan | investment | other, 필수)
- `starting_balance_cents` (정수, 필수, 기본값 0)
- `is_archived` (boolean, 기본값 false)

**카테고리(Category) 필드**:
- `id` (UUID, 필수)
- `name` (문자열, 필수, 최대 100자)
- `parent_id` (UUID, 선택) // 하위카테고리용
- `icon` (문자열, 선택)
- `color` (hex 문자열, 선택)

### 데이터 정확도 요구사항

| 필드 | 정확도 | 검증 |
|------|--------|------|
| 금액 | 센트 단위 정확 | 정수, 반올림 오류 없음 |
| 날짜 | 일(day) 단위 정확 | ISO 8601, 사용자 타임존 |
| 카테고리 | 사용자 정의 | ID 존재 검증 |
| 계정 | 사용자 정의 | ID 존재 검증 |

### 데이터 보관 정책

- **v1**: 자동 삭제 없음
- **향후**: 사용자 설정 아카이빙(예: 3년 이상)

## Non-Goals

v1에서 명시적으로 **포함하지 않는 기능들**

| 기능 | 이유 | 향후 버전 |
|---------|--------|----------------|
| 은행 연동 | 프라이버시 우려, 복잡성 | v2+ (선택) |
| 클라우드 동기화 | 오프라인 우선 | v2 (선택) |
| 이메일/인증 | 진입 장벽 최소화 | v2 (가족 기능용) |
| 다중 사용자/가족 공유 | v1 범위 제한 | v2 |
| 거래 수정 | 단순성 유지 | v2 |
| 반복 거래 | 복잡성 | v2 |
| 청구서 알림 | 범위 외 | v2+ |
| 투자 추적 | 가치 평가 로직 복잡 | v2+ |
| 세금 카테고리 | 회계 전문성 필요 | v3+ |
| 영수증 OCR | ML/클라우드 필요 | v2+ |
