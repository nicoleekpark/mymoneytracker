## 제품 관련 열린 질문

### Q1: 최종 제품명

**현재**: "HoH Finance Tracker" (임시 이름)

**옵션**:
- HoH Finance
- HoH Wallet
- CashFlow
- Manual Money
- Intentional Finance

**결정 필요 시점**: v1 개발 종료 전 (공개 릴리스 이전)

**이해관계자**: 제품, 마케팅, 법무(상표권 확인)

---

### Q2: Cash Flow 탭의 목적

**배경(Context)**: 대시보드는 4가지 모드가 있음  
Overview, **Cash Flow**, Accounts, Net Worth

**문제**: Cash Flow 탭이 현재 플레이스홀더 상태다  
Overview와 무엇을 다르게 보여줘야 할까?

**옵션**:
1. **상세 수입/지출 흐름 다이어그램** (Sankey 차트 스타일)
2. **시간에 따른 순현금 포지션** (라인 그래프)
3. **지출 추세 기반 예상 현금 소진 기간(runway)**
4. **탭 자체 제거** — Cash Flow는 이미 Overview에 포함됨

**논의**:
- Overview에는 이미 일별 현금 흐름 캘린더가 있음
- 사용자는 "Cash Flow"에서 더 기술적이고 깊은 분석을 기대할 수 있음
- "Trends" 또는 "Analysis"로 이름 변경 고려 가능

**결정 필요 시점**: v1.1 릴리스 전

---

### Q3: 카테고리별 예산 (Per-Category Budgets)

**배경**: v1은 전체 월 예산만 지원

**사용자 요청**:  
"Food는 $500, Transport는 $200 이런 식으로 설정하고 싶어요"

**옵션**:
1. **v1에 추가**: 범위 증가, 출시 지연
2. **v2까지 대기**: v1 단순성 유지
3. **하이브리드**: 설정은 가능하지만 강제하지 않음 (패시브 트래킹)

**트레이드오프**:
| 옵션 | 장점 | 단점 |
|------|------|------|
| v1 추가 | 파워 유저 만족 | 복잡도 증가, UI 혼잡 |
| v2 대기 | v1 집중도 유지 | 일부 사용자 실망 |
| 하이브리드 | 구현 부담 적음 | 반쪽짜리 기능으로 혼란 |

**현재 방향**: v2까지 대기

**결정 필요 시점**: 2026년 2월

---

### Q4: 예산 초과 시 행동

**배경**: 사용자가 월 예산을 초과했을 때

**현재 동작**:
- 예산 초과 날짜 표시
- 진행 바가 빨간색으로 변경

**질문**: 추가 가이드를 제공해야 할까?

**아이디어**:
- 예산 초과 시 알림 전송
- 행동 제안 ("카테고리 X 지출을 잠시 중단하세요")
- 현재 추세 기반 월말 예상 지출 표시
- 월 중 예산 증액 허용 (권장되지 않음)

**추가 리서치 필요**:  
사용자는 처방적 가이드를 원하는가, 아니면 인지만으로 충분한가?

## 기술 관련 열린 질문

### T1: 데이터베이스 오프라인 동기화/백업 전략 (v1)

**배경**: v1은 오프라인 전용, 클라우드 동기화 없음

**문제**: 사용자는 재무 데이터를 어떻게 백업할까?

**옵션**:
1. **수동 내보내기(Manual Export)**:
   - "데이터 내보내기" → JSON/CSV 다운로드
   - 장점: 단순, 사용자 통제
   - 단점: 잊기 쉬움, 자동 백업 아님
2. **기기 시스템 백업만 사용**:
   - iOS/Android 시스템 백업 의존
   - 장점: 구현 비용 0
   - 단점: 시스템 백업 미활성 사용자 존재
3. **둘 다 제공**:
   - 수동 내보내기 + 시스템 백업 안내
   - 장점: 가장 안전
   - 단점: 구현 작업 증가

**현재 방향**: 옵션 3

**결정 필요 시점**: v1 릴리스 전

**구현 비용**: 내보내기 기능 약 2일

---

### T2: 마이그레이션 테스트 전략

**배경**: 커스텀 SQLite 마이그레이션 시스템 사용

**문제**: 마이그레이션이 기존 사용자 데이터를 깨지 않는지 어떻게 보장할까?

**현재 방식**:
- 시뮬레이터 수동 테스트
- 자동화된 마이그레이션 테스트 없음

**제안**:
- 각 스키마 버전별 테스트 DB 스냅샷 생성
- CI에서 모든 마이그레이션 forward/backward 실행
- 마이그레이션 후 데이터 무결성 검증

**블로커**: 테스트 인프라 구축 필요

**결정 필요 시점**: 다음 마이그레이션 전 (우선순위 높음)

---

### T3: 대용량 데이터셋 성능

**배경**: 현재 약 100건 거래로만 테스트됨

**질문**: 거래 10,000건 이상에서 성능은 어떨까?

**우려 사항**:
- 월별 대시보드 쿼리 (해당 월 전체 스캔)
- 거래 리스트 스크롤 성능
- 카테고리 분해 계산

**필요 테스트**:
- 1만 건 이상 거래 시드 스크립트 생성
- 쿼리 성능 벤치마킹
- 필요 시 DB 인덱스 추가

**결정 필요 시점**: v1 공개 릴리스 전

---

### T4: 웹 최적화

**배경**: v1은 웹 지원하지만 모바일 우선

**질문**: 데스크톱 웹 UX를 최적화해야 할까?

**아이디어**:
- 와이드 화면에서 2컬럼 레이아웃
- 키보드 단축키 (화살표로 날짜 이동)
- 대량 거래 입력 (CSV import)
- 데스크톱용 날짜 선택 UI (iOS 휠 아님)

**트레이드오프**:
- 노력: 약 1–2주
- 이점: 웹 사용성 개선
- 리스크: 모바일/웹 코드 분기 증가

**현재 방향**: v1은 모바일 우선 유지, v1.5에서 웹 최적화

## UX 관련 열린 질문

### U1: 첫 사용자 경험 (FTUE)

**배경**: 현재 온보딩 튜토리얼 없음

**질문**: 얼마나 많은 가이드를 제공할까?

**옵션**:
1. **전체 튜토리얼 (3–5 화면)**:
   - "HoH에 오신 것을 환영합니다"
   - "첫 거래 추가"
   - "예산 설정"
   - "대시보드 탐색"
   - 장점: 친절
   - 단점: 스킵 가능성 높음, 유지 비용
2. **인라인 힌트**:
   - 첫 대시보드: "여기를 눌러 거래 추가"
   - 캘린더: "날짜를 탭하면 거래 확인"
   - 장점: 맥락적
   - 단점: 놓치기 쉬움
3. **명확한 빈 상태 메시지**:
   - "아직 거래가 없습니다. +를 눌러 첫 지출을 추가하세요"
   - 장점: 단순
   - 단점: 사용자 주도성 가정

**현재 구현**: 옵션 3

**결정 필요 시점**: 베타 피드백 이후

---

### U2: 커스텀 카테고리 관리

**배경**: v1은 고정 카테고리 목록 사용

**사용자 요청**: "내 카테고리를 직접 추가하고 싶어요"

**옵션**:
1. **완전 자유 커스터마이즈**:
   - 카테고리 추가/수정/삭제
   - 장점: 유연성 극대화
   - 단점: 혼란 가능
2. **상위 카테고리 고정 + 하위만 커스텀**:
   - 균형점
3. **v1에서는 미지원**:
   - 사용자 피드백 대기

**현재 방향**: v1은 옵션 3, v2는 옵션 2

---

### U3: 거래 상세 화면 동작

**배경**: 현재 거래 상세는 읽기 전용

**질문**: v1에서 바로 수정 허용할까?

**우려**:
- 날짜 변경 시 예산 계산 영향
- 계정 잔액 연쇄 업데이트
- 감사 로그 필요성

**옵션**:
1. **읽기 전용 (v1)**  
2. **v2에서 전체 수정 + 감사 로그**
3. **삭제만 허용 (v1)**

**현재 방향**: 옵션 3
